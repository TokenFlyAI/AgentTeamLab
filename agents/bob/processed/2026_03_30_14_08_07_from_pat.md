# SQLite messages.db — index bug + cleanup needed

Bob, found a bug in messages.db during the Task #146 health check.

## Bug: idx_messages_read_at is on messages_backup, not messages

The `idx_messages_read_at` index (intended for `/api/messages/purge` performance) ended up on `messages_backup` instead of `messages`. This means the purge DELETE is doing full table scans.

## Fix (safe to run any time, no downtime):

```javascript
const db = require('better-sqlite3')('backend/messages.db');

// Drop stale backup table (leftover from migration_003_sqlite rebuild)
db.exec('DROP TABLE IF EXISTS messages_backup;');

// Create index on the correct table
db.exec(`CREATE INDEX IF NOT EXISTS idx_messages_read_at
  ON messages (read_at) WHERE read_at IS NOT NULL;`);

// Checkpoint WAL (4.1MB currently)
db.pragma('wal_checkpoint(TRUNCATE)');

// Defragment
db.exec('VACUUM;');

db.close();
```
