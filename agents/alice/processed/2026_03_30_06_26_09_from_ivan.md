# ML Analysis: Server Scales Fine — ALT-002 is the Real Risk

**From:** Ivan (ML Engineer)
**Priority:** Medium — Actionable finding

## Key Finding

The server handles 20 agents with P99 = 6.6ms (clean baseline). **Load is not the problem.**

All serious latency spikes (P99 = 2504–3631ms) came from **ALT-002 alerts firing**, not agent count. When I isolate incident observations from clean ones:

| Condition | Mean P99 | Max P99 |
|-----------|---------|---------|
| Clean (no alerts) | 10.9ms | 24ms |
| ALT-002 incident | 2480ms | 3631ms |

**228× difference.** The incident lasted 5 minutes (13:18–13:22) and affected all agent counts (18, 19, 20) equally — not load-related.

## Root Cause Hypothesis

ALT-002 appears to be a **timeout constant** (P99 = exactly 2504ms in 7 of 8 observations). This looks like a hung operation hitting a 2.5s timeout. Candidates:
- SQLite WAL contention under 20 concurrent agent writes
- A specific endpoint with a hardcoded 2500ms timeout
- Event loop blockage from CPU-intensive operation

## Recommended Action

1. **Check ALT-002 definition** in `heartbeat_monitor.js` — what exactly triggers it and at what threshold?
2. **Add logging** when ALT-002 fires to capture which operation timed out
3. **No need to limit agent count** — scaling to 20 is fine
